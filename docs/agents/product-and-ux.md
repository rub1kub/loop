# Продукт и UX

## Суть LOOP

Главная эмоция: «я участвую в живом цикле». Пользователь должен понимать происходящее без
знаний о блокчейне, но видеть, что важный результат подтверждён TON.

LOOP не прячет риски за атмосферой. Загадочность создаётся образом банки, движения песка,
короткими переходами и строгой типографикой; правила, сумма, комиссия, срок и невозможность
гарантировать выплату сообщаются прямо.

## Поверхности продукта

| Поверхность             | Аудитория         | Авторизация               | Назначение                         |
| ----------------------- | ----------------- | ------------------------- | ---------------------------------- |
| Telegram Mini App       | пользователь      | signed Telegram initData  | BANK, DUEL, рейтинг, профиль       |
| Telegram inline message | пользователь/друг | Telegram                  | передача прямого вызова DUEL       |
| `/control`              | владелец          | одноразовый TON proof     | приложение и контракты             |
| browser mock            | разработчик       | `VITE_MOCK_TELEGRAM=true` | UI, E2E и скриншоты без транзакций |

Обычный consumer-сайт с browser login сейчас **не реализован**. Корень публичного домена
ожидает Telegram initData, а `/control` — адрес владельца. Превращение пользовательской части в
самостоятельный сайт потребует отдельного решения по identity, anti-abuse, социальным ссылкам и
TON proof; это не следует делать скрытой fallback-авторизацией.

## BANK простыми словами

BANK — общая FIFO-очередь выплат.

1. Пользователь выбирает взнос от `1` до `100 GRAM`.
2. Выбирает цель `1.25×`, `1.5×` или `2×`.
3. Внешний кошелёк подписывает создание позиции.
4. Из каждого нового principal контракт отправляет комиссию BANK в treasury.
5. Остаток сначала наполняет самую старую незавершённую позицию, затем следующую.
6. Когда старая позиция достигает цели, контракт автоматически отправляет её владельцу полную
   целевую выплату.
7. Оставшаяся после старой очереди сумма становится начальным наполнением новой банки.
8. Новая позиция ждёт будущие взносы. Если они прекращаются, она может не завершиться никогда.

Это не схема «B всегда полностью платит A». Один взнос может частично наполнить A, завершить
несколько старых позиций или оставить остаток новой позиции B.

### Числовой пример

При пустой очереди A вносит `1 GRAM` и выбирает `1.5×`:

- цель A: `1.5 GRAM`;
- комиссия `1%`: `0.01 GRAM`;
- `0.99 GRAM` сразу становится наполнением банки A;
- осталось наполнить `0.51 GRAM`.

Позже B вносит `1 GRAM`:

- `0.01 GRAM` уходит как комиссия;
- первые `0.51 GRAM` завершают A, и контракт отправляет A `1.5 GRAM`;
- оставшиеся `0.48 GRAM` начинают наполнять позицию B;
- B ждёт следующих участников.

Округление всегда целочисленное вниз в nanoGRAM. Газ прикладывается отдельно и не входит в
principal.

### Состояния BANK

| API state              | Что видит пользователь                               |
| ---------------------- | ---------------------------------------------------- |
| `pending_confirmation` | ждём появление подписанной транзакции в TON          |
| `queued`               | позиция подтверждена, но ещё не получила наполнения  |
| `partially_funded`     | песок и процент плавно отражают прогресс             |
| `completed`            | промежуточная проекция перед подтверждённой выплатой |
| `payout_sent`          | выплата подтверждена, доступна ссылка на TON         |
| `failed`               | intent истёк или не прошёл проверку                  |

### UX BANK

- Банка, а не баланс, занимает центральное место.
- Уровень песка равен `progress_bps / 10_000`; изменение должно быть медленным и плавным.
- Поверхность песка слегка движется волной, но не создаёт ложного прогресса.
- Показываются читабельное место в незавершённой очереди, число активных участников, осталось до
  цели и последнее подтверждение.
- Создание позиции — полноэкранный пошаговый сценарий: сумма → цель → подтверждение → ожидание.
- Обязательно понятно сообщить: срока нет, досрочной отмены после подтверждения нет, выплата
  зависит от новых взносов.

## DUEL простыми словами

Новый DUEL — равный вызов `50/50`:

1. Первый участник выбирает свою ставку.
2. Приложение создаёт локальный секрет и commitment, связанный с сетью, контрактом, offer ID и
   адресом кошелька.
3. Внешний кошелёк блокирует ставку в `DuelEscrow`.
4. Второй участник блокирует такую же ставку.
5. После матча оба раскрывают секреты.
6. Контракт вычисляет результат из двух заранее зафиксированных секретов.
7. Победитель получает общий пул минус `2.5%`.

Для новых равных offers минимальный контрактный pool — `1 GRAM`, поэтому эффективная
минимальная ставка каждого участника — `0.5 GRAM`. Максимальный pool — `100 GRAM`, то есть
максимальная равная ставка — `50 GRAM`.

### AFK-поиск

- Пользователь создаёт on-chain offer и может закрыть Mini App.
- API резервирует самый старый совместимый `50/50` offer с тем же pool через
  `FOR UPDATE SKIP LOCKED`.
- Резерв действует пять минут. Если второй funding не финализируется, AFK-offer снова открыт.
- До матча владелец может отменить offer; после expiry любой участник может вызвать refund.

### Прямой вызов

- Creator сначала открывает `OpenDirectOffer` с независимым 256-bit invite ID.
- После финализации кнопка «Пригласить в Telegram» вызывает inline mode с точным offer ID.
- Telegram-сообщение содержит только отображение проверенных условий и кнопку «ПРИНЯТЬ».
- Получатель проходит Telegram auth, подтверждает внешний адрес и резервирует приглашение.
- API выдаёт короткий Ed25519 permit, связанный с network, contract, invite ID, creator offer,
  invited wallet и сроком.
- `AcceptDirectOffer` проверяет permit в контракте и атомарно связывает два адреса.
- Другой адрес не может украсть или повторно принять вызов.

### Commit–reveal и recovery

| Ситуация                           | Результат контракта               |
| ---------------------------------- | --------------------------------- |
| оба раскрыли вовремя               | детерминированный weighted winner |
| раскрыл только один                | после deadline побеждает он       |
| никто не раскрыл                   | оба principal возвращаются        |
| unmatched offer отменён владельцем | principal возвращается            |
| unmatched offer истёк              | permissionless refund             |

Пауза запрещает новые offers и matches, но не должна блокировать cancel, expiry, reveal или
settlement.

### Состояния DUEL

Offers: `pending_funding → open → reserved → matched → settled/refunded`.
Боковые терминальные ветки: `cancelled`, `expired`, `rejected`.

Duels: `revealing → settled/refunded/expired`.

Секрет DUEL хранится в Telegram SecureStorage. Backend его не получает. В mock-режиме допустим
только `sessionStorage`.

## Рейтинг

Рейтинг пересчитывается из подтверждённых строк текущего календарного месяца; отдельной
изменяемой таблицы score нет.

```text
+100  подтверждённая выплата BANK
 +60  подтверждённое завершение DUEL
 +20  раскрытие DUEL вовремя
 +25  квалифицированный друг
 -40  пропущенное раскрытие
```

Минимум — ноль.

| Уровень | Score за месяц |
| ------- | -------------: |
| SIGNAL  |          0–199 |
| PULSE   |        200–499 |
| ORBIT   |        500–999 |
| LOOP    |         1,000+ |

Сортировка: score, затем число подтверждений. «Мой круг» включает пользователя и обе стороны
квалифицированных referral edges. Live pulse показывает активных участников BANK/DUEL и число
подтверждений за 24 часа, но не влияет на score.

Нельзя добавлять в формулу сумму, прибыль, баланс, победы или поражения без явного пересмотра
продуктового принципа.

## Профиль, рефералы и PLUSH BRICK

- Профиль показывает Telegram identity, статистику BANK/DUEL и уровень LOOP.
- Внешний кошелёк можно подключить/отключить через TON Connect; приложение средств не хранит.
- Реферальный `startapp=ref_<code>` записывается один раз, self-referral и повторная атрибуция
  отвергаются.
- Текущий worker квалифицирует друга после подтверждённого DUEL settlement. BANK payout пока не
  запускает qualification, хотя общий продуктовый текст говорит о «подтверждённом действии».
- PLUSH BRICK проверяется отдельно в mainnet как Jetton ownership proof.
- Скидка PLUSH BRICK сейчас выключена: testnet-контракт не может безопасно применять
  mainnet-зависимую комиссию без дополнительного proof/oracle design.

Сейчас профильный `ReferralReward.reward_points` начисляет `100` за DUEL-квалификацию, а
формула месячного рейтинга использует `+25`. Эти числа относятся к двум разным существующим
проекциям, но в интерфейсе называются одинаково «очки». До продуктового решения их нельзя
молча объединять.

## Onboarding

Onboarding из трёх историй объясняет:

1. LOOP = BANK + DUEL, внешний кошелёк остаётся внешним;
2. BANK = очередь, срока и досрочной отмены нет;
3. DUEL = равные условия, а подтверждённые действия формируют LOOP Score.

Он показывается только при `onboarding_enabled=true` и `onboarding_seen=false`. После завершения
`onboarding_seen` сохраняется на сервере. В профиле есть явный replay.

## Telegram presentation

- SDK загружается асинхронно; desktop может взять `tgWebAppData` из исходного launch URL.
- `ready()`, `expand()`, BackButton, haptics, safe-area events и SecureStorage используются
  feature-detection, без предположения о полной реализации bridge.
- MainButton скрыт: основные действия остаются частью монохромного интерфейса LOOP.
- Native `android`, `android_x`, `ios` запрашивают fullscreen при Telegram `>=8.0`.
- Desktop/macOS/web выходят из fullscreen и остаются expanded.
- BotFather Main App должен быть **Fullsize**. Мобильный fullscreen выбирает сам клиентский код.
- Header/background/bottom bar переустанавливаются в `#000000` после `ready`, theme,
  activation и fullscreen events.

### Viewport и клавиатура

`apps/web/src/viewport.ts` хранит stable height, защищает максимальный верхний inset сессии,
отслеживает `visualViewport`, Telegram safe areas и focus transitions. CSS использует:

- `--loop-stable-height`;
- `--loop-visual-height`;
- `--loop-visual-page-top`;
- `--loop-safe-area-inset-*`;
- корневой класс `.keyboard-open`.

При исправлении клавиатуры нельзя:

- делать глобальный `scrollIntoView`, который сдвигает весь Mini App;
- сбрасывать верхний inset при кратком изменении Telegram viewport;
- возвращать tab bar между `pointerup` и `click`;
- одновременно позиционировать sheet от stable viewport и от уже уменьшенного visual viewport;
- проверять только Chrome desktop вместо iOS WebKit.

## Визуальные и текстовые правила

- Цвета: black/white/gray, контрастные основные действия.
- Сетки и разделители используются экономно; один смысловой блок не режется множеством линий.
- Input должен выглядеть как input: видимая рамка/фон, caret, label и состояние focus.
- Выпадающий элемент имеет chevron или иной явный disclosure indicator.
- Микроанимация не мешает вводу; поддерживается `prefers-reduced-motion`.
- Выделение текста, context menu, drag изображений и zoom gestures заблокированы внутри Mini App.
- Заголовки могут быть короткими и атмосферными, но действие и риск формулируются буквально.
- Не использовать wallet-first copy: «твой баланс», «храни средства», «портфель».
- Не использовать casino copy: «азарт», «сорви куш», «проигрыш».
- Не обещать срок, доход, ликвидность или безопасность, которых контракт не гарантирует.

## Известные продуктовые вопросы

1. Consumer browser mode не реализован, несмотря на ранее обсуждавшееся желание иметь обычный
   сайт.
2. UI предварительно проверяет минимальную ставку DUEL как `0.25 GRAM`, тогда как эффективный
   минимум равного offer по контракту — `0.5 GRAM`; backend корректно отвергает меньший pool.
3. Referral reward `100` и rating referral score `25` выглядят как одна метрика, хотя считаются
   разными путями.
4. Referral qualification сейчас вызывается только DUEL settlement, а не любым BANK/DUEL
   подтверждением.
5. BANK не имеет cancel/early refund после подтверждения; это продуктовый риск, а не дефект UI.
6. Рейтинг формирует социальный статус, но не имеет отдельной серверной сезонной фиксации:
   формула пересчитывается из событий текущего месяца.
