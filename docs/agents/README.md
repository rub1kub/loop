# LOOP: база знаний для агентов

Это каноническая точка входа для агента, который впервые работает с LOOP.

репозитория. Детали разнесены по связанным разделам, чтобы не восстанавливать контекст из

## Модель продукта за минуту

LOOP — социальное приложение внутри Telegram с внешним TON-кошельком. Это не кошелёк, не
портфель и не внутренний баланс.

- **BANK** — очередь позиций FIFO. Каждый новый взнос после комиссии наполняет самые ранние
  незавершённые позиции. Выплата происходит автоматически при достижении цели. Новые взносы
  могут прекратиться, поэтому срок и выплата не гарантированы.
- **DUEL** — равный вызов двух участников. Новый публичный продукт создаёт только условия
  `50/50`: оба блокируют одинаковую сумму, а контракт завершает commit–reveal и платит
  победителю общий пул за вычетом комиссии.
- **РЕЙТИНГ** — месячная репутация за подтверждённые завершения, своевременные раскрытия и
  квалифицированных друзей. Размер ставки, прибыль, баланс, победы и поражения не повышают
  статус.
- **ПРОФИЛЬ** — Telegram-личность, внешний TON Connect, история подтверждений, рефералы и
  проверка PLUSH BRICK.
- **ПАНЕЛЬ УПРАВЛЕНИЯ** — отдельный обычный браузерный интерфейс владельца. Он не использует
  Telegram-авторизацию и не является частью Mini App.

## Неприкосновенные инварианты

### Продукт

1. LOOP не создаёт пользовательские кошельки, не принимает seed-фразы и не ведёт внутренний
   доступный баланс.
2. TON Connect нужен только для доказательства владения адресом и подписания транзакции внешним
   кошельком.
3. BANK и DUEL полностью разделены: разные контракты, таблицы, события, история, комиссии и
   экраны.
4. Новые DUEL через приложение всегда `50/50`. Поддержка `25/75` и `75/25` в контракте оставлена
   только для совместимости и восстановления старых предложений.
5. Рейтинг не поощряет финансовый объём, прибыль, победы или богатство.

### Доверие и деньги

1. Ответ кошелька «транзакция отправлена» не доказывает финансовое действие.
2. Источник истины — успешная транзакция текущего контракта с включением в masterchain.
3. PostgreSQL — идемпотентная проекция цепочки и продуктовые данные, а не кастодиальный ledger.
4. Redis хранит только временные блокировки, nonce, rate limits и оперативные метрики.
5. Клиент не определяет сумму, условия, результат, очередь, комиссию или финальность.
6. Нельзя переносить средства, события или идентификаторы между BANK и DUEL.
7. Mainnet-транзакции блокируются конфигурацией и кодом.

### Интерфейс

1. Основная палитра — чёрный, белый и серый; без неона, криптоградиентов и синей основной кнопки.
2. Банка — главный объект BANK; баланс и кошелёк не становятся центром экрана.
3. Технические слова не выносятся в основной пользовательский текст. Вместо «ончейн-пруф»
   используется «подтверждение в TON», вместо инженерного `testnet` — «тестовые GRAM» только
   там, где это нужно для безопасности.
4. Обучающие подсказки показываются один раз; их можно повторить из настроек.
5. На iOS/Android приложение запрашивает fullscreen, на desktop/macOS/web остаётся в обычном
   expanded/fullsize-окне Telegram.
6. Telegram header, background и bottom bar всегда возвращаются к `#000000`.
7. При открытой клавиатуре tab bar скрыт, верхняя safe area сохраняется, корневой viewport не
   должен прокручиваться или сдвигаться.

### Репозиторий и эксплуатация

1. `sources/` — синхронизированный read-only контекст; его нельзя менять.
2. Никогда не читать, копировать или коммитить `.env*`, `wallets.toml`, mnemonic, private key,
   bot token, API key, SSH credential, Telegram `initData` или wallet proof.
3. Контрактное финансовое правило нельзя «исправить» только в API. Для изменения поведения
   нужен новый проверенный контракт и новый manifest.
4. Обычный релиз приложения не разворачивает контракт.
5. Любое утверждение о живой сети проверяется заново непосредственно перед действием.

## Иерархия источников истины

При расхождении используй следующий приоритет:

1. Живое состояние TON для баланса, `locked`, `paused`, owner, treasury и текущих транзакций.
2. Текущий Tolk-код и его локальная сборка для правил контракта.
3. Миграции и текущие модели/маршруты для схемы приложения.
4. `deployments/testnet/*.json` для адреса, хеша кода и доказательства развёртывания.
5. Эта база знаний и профильные документы `docs/*.md`.
6. `README.md`, `CHANGELOG.md`, `progress.md` и дизайн-отчёты как обзор или история.
7. История чата и старые вложения — только как контекст намерений, не как описание текущего кода.

Манифест содержит снимок конфигурации при развёртывании. Поля `locked`, баланс, пауза, owner,
treasury и fee могут меняться; не считай их живыми без getter.

## Термины

| Термин             | Значение                                                                  |
| ------------------ | ------------------------------------------------------------------------- |
| GRAM               | единица, используемая текущим тестовым TON-окружением LOOP                |
| nanoGRAM           | `1 GRAM = 1_000_000_000 nanoGRAM`                                         |
| позиция BANK       | один активный FIFO-запрос внешнего адреса на целевую выплату              |
| principal          | исходный взнос позиции BANK                                               |
| target payout      | `principal × multiplier`, округлённый вниз в nanoGRAM                     |
| offer DUEL         | заблокированная ставка и commitment одного участника                      |
| duel               | пара совместимых offers с общим reveal deadline                           |
| commit–reveal      | схема, где секрет фиксируется хешем до матча и раскрывается после         |
| direct permit      | короткая Ed25519-подпись LOOP, привязанная к адресу приглашённого         |
| projection         | проверенное отображение TON-события в PostgreSQL                          |
| finalized          | успешная транзакция с положительным masterchain sequence                  |
| LOOP Score         | вычисляемая месячная репутация без финансового объёма                     |
| application intake | off-chain переключатели maintenance/BANK/DUEL в панели                    |
| contract pause     | on-chain запрет новых входов; пользовательские recovery paths сохраняются |

## Перед любой правкой

1. Уточни, какой bounded context меняется: BANK, DUEL, social/rating, Telegram или control.
2. Открой только соответствующие файлы из [agent-playbooks.md](agent-playbooks.md).
3. Проверь, не меняет ли задача финансовое правило, публичный API, auth, БД или контракт.
4. Сохрани разделение BANK/DUEL и доказательную границу chain worker.
5. Запусти минимальный набор проверок из [operations.md](operations.md).
6. Обнови эту базу, если изменился инвариант, endpoint, таблица, контракт, runbook или известный
   риск.
